>## 💡 项目背景与解决痛点

在传统的移动端大模型（LLM）交互中，通常面临**输入效率低（仅文本）**以及**响应等待时间长（网络抖动、首字响应慢）**的痛点。
本项目基于 Taro + React 架构，为公司 AI Agent 产品构建了高可靠的**多模态交互系统**，重点解决了弱网环境下的流式传输体验与实时语音对讲的低延迟问题。

---

## 🛠️ 核心架构与 AI 工程化实践

作为本项目的核心前端开发，我不仅负责 UI 渲染，更重要的是作为 Agent 的“感知与表达器官”，构建了匹配大模型能力的流式交互架构：

### 1. ⚡ 打通多模态交互链路 (低延迟语音)
* **痛点**：传统高封装 SDK 灵活性差，无法满足 Agent 实时交互需求。
* **方案**：弃用传统 SDK，直接基于 **WebSocket** 推送音频数据流。使用 `WebAudio API` 接管前端音频上下文，在前端完成基础的音频格式转换（`Int16` 与 `Float32` 映射），配合 Agent 后端的实时语音识别，实现了**“边录边转写”**的极低延迟体验。

### 2. 🌊 大模型长文本流式渲染 (Streaming & TTFT 优化)
* **痛点**：移动端网络抖动会导致 LLM 吐字卡顿，用户感知的首字响应时间 (TTFT) 过长。
* **方案**：
  * **底层支持**：自主封装 SSE (Server-Sent Events) 解析器，实现长文本流式分块接收。
  * **双缓冲平滑策略**：设计前端双缓冲队列策略，平滑大模型的吐字速率。消除网络抖动带来的视觉卡顿，实现极度丝滑的“打字机效果”，**将用户感知的 TTFT 降低了约 40%**。

3. 🛡️ 解决异步登录竞态与 Token 无感刷新
设计“全局事件总线 + 生命周期守卫”机制，彻底解决多组件并发请求时的异步登录竞态问题。

封装 Axios 适配器实现并发 401 拦截锁与 Token 无感刷新，保障 Agent 对话过程不被中断，极大提升用户体验。

4. 🪟 小程序双线程稳定性治理
针对小程序独特的双线程架构，引入**“销毁模式”**，严格拦截页面卸载后的遗留异步回调。

彻底解决了由于长轮询或 WebSocket 未及时清理导致的内存泄露与崩溃问题，保障了长会话场景的稳定性。
